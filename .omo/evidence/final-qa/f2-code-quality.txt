ISSUE F2-1 [P1 — swallowed error, functional concern]
  Location: Export() lines 103-108
  Code:
    msgs, msgErr := p.reader.listMessages(string(sessionID))
    if msgErr == nil {
        for _, hm := range msgs {
            result.Messages = append(result.Messages, mapMessage(hm))
        }
    }
  Problem: If listMessages fails, msgErr is silently discarded. The caller
  receives a session struct with zero messages and no indication anything
  went wrong. A corrupt DB page or schema mismatch would be invisible.
  Fix: propagate msgErr (or at minimum log it at warn level).

ISSUE F2-2 [P1 — swallowed error, functional concern]
  Location: Export() lines 111-114
  Code:
    children, err := p.loadChildSessions(string(sessionID), mode)
    if err == nil && len(children) > 0 {
        result.Children = children
    }
  Problem: Same pattern — loadChildSessions error is dropped. The session
  is returned without children and no error surfaced.
  Note: err is reused from the earlier p.reader.readSession call (line 93).
  The re-assignment on line 111 shadows the earlier check. Both the
  shadowing and the silent drop are worth fixing together.
  Fix: `if err != nil { return nil, fmt.Errorf("hermes: loading children: %w", err) }`

ISSUE F2-3 [P2 — resource leak]
  Location: New() line 32-34, no corresponding close anywhere
  Code:
    if dbr, err := newDBReader(hermesHome); err == nil {
        p.reader = dbr
    }
  Problem: The *sql.DB opened inside newDBReader is never closed.
  Provider has no Close() method and does not implement io.Closer.
  In the CLI use-case this is benign (process exits), but in the long-
  running `aisync serve` path the connection leaks for the server lifetime.
  Fix: Add Close() error method to Provider; call p.reader.close() from it;
  have the server defer provider.Close() at startup.

ISSUE F2-4 [P3 — minor]
  Location: lines 38-41, 138-142, 144-147
  Trivial doc comments that restate the signature:
    "// Name returns the provider identifier."
    "// CanImport reports that Hermes does not support session import."
    "// Import always returns an error because Hermes is read-only."
  Matches existing codebase style (opencode adapter has the same pattern),
  so not a blocking concern — noted for consistency.
